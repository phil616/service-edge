# 发版指南（GitHub Tag 自动发版）

云梦镜像边缘服务网络（service-edge）通过 **推送 Git tag** 触发 GitHub Actions 自动构建并发布 Release。本文档说明完整流程，便于手动发版。

> 仓库：`phil616/service-edge`（GitHub 远程名为 `origin`）。
> 工作流定义见 [`.github/workflows/release.yml`](./.github/workflows/release.yml)。

---

## 一、工作原理

向 `origin` 推送一个名称匹配 `v*` 的 tag（如 `v1.2.3`），即触发 `Release` 工作流，自动完成：

1. 检出代码、安装 Go（按 `go.mod`）与 Node 20；
2. `make web` 构建前端并内嵌进后端二进制（`go:embed`）；
3. 交叉编译四个产物（注入版本号 `-X main.version=<tag>`）：
   - `service-edge_linux_amd64` / `service-edge_linux_arm64`（控制面）
   - `agent_linux_amd64` / `agent_linux_arm64`（Agent）
4. 生成 `SHA256SUMS` 校验和；
5. 为每个二进制生成 **SLSA 构建溯源**（`actions/attest-build-provenance`）；
6. 用 `gh release create --generate-notes` 创建 Release 并上传以上文件，发布说明自动生成。

> Agent 产物命名 `agent_linux_<arch>` 与安装脚本 `${下载基址}_linux_<arch>` 的拼接规则一致，可直接把 Release 的 `latest` 下载地址作为 Agent 下载源（见「系统设置 → Agent 下载设置」）。

---

## 二、前置条件

- 本机已安装并登录 GitHub CLI：
  ```bash
  gh auth status        # 应显示 Logged in to github.com（账号 phil616）
  ```
  若未登录：`gh auth login`（在交互式终端执行，可在对话中输入 `! gh auth login`）。
- 对 `origin` 有推送权限。
- 待发布的代码 **已提交并推送到 `origin/main`**（tag 指向的提交必须包含上述工作流文件）。

---

## 三、版本号规范（语义化版本 SemVer）

格式 `vMAJOR.MINOR.PATCH`：

| 变更类型 | 递增位 | 示例 |
|---|---|---|
| 仅修复 Bug，无新功能/不破坏兼容 | PATCH | `v1.1.1 → v1.1.2` |
| 新增向后兼容的功能 | MINOR（PATCH 归零） | `v1.1.2 → v1.2.0` |
| 不兼容的重大变更 | MAJOR（其余归零） | `v1.2.0 → v2.0.0` |

查看现有 tag 以确定下一个版本号：
```bash
git tag --sort=-v:refname | head        # 本地
git ls-remote --tags origin             # 远端（GitHub）
```

## 四、手动发版步骤

### 1. 确认代码已就绪并与 GitHub 同步

```bash
# 工作区应为干净状态；如有改动需先提交
git status

# 拉取远端 tag/分支信息（只读）
git fetch origin --tags

# 确认本地 HEAD 与 origin/main 一致（输出应为空，表示没有未推送的提交）
git log --oneline origin/main..HEAD
```

- 若 `git status` 有未提交改动 → 先 `git add <文件>` 并 `git commit`。
- 若上面 `git log` 有输出（本地领先 origin）→ 先 `git push origin main`，否则 **本次发布不会包含这些提交**。

> 本项目还有一个镜像远程 `cnb`（cnb.cool）。发版工作流仅在 GitHub 生效，**tag 只需推送到 `origin`**。如需保持镜像分支同步，可额外 `git push cnb main`（但不必推送 tag 到 cnb）。

### 2. 创建带注释的 tag

使用带注释（annotated）tag，并在消息里写清本次变更：

```bash
git tag -a v1.2.0 -m "$(cat <<'EOF'
v1.2.0 — 云梦镜像边缘服务网络 (service-edge)

主要更新：
- <一句话描述变更 1>
- <一句话描述变更 2>
EOF
)"
```

确认 tag 内容与指向：
```bash
git show -s --format='%H' v1.2.0     # tag 指向的提交
git tag -n9 v1.2.0                   # 查看 tag 说明
```

### 3. 推送 tag 到 GitHub（触发发版）

```bash
git push origin v1.2.0
```

### 4. 观察工作流执行

```bash
# 列出最近的 Release 工作流运行
gh run list --workflow=release.yml --limit 3

# 实时跟踪某次运行直到结束（RUN_ID 取自上面的列表）
gh run watch <RUN_ID> --exit-status
```

约 2 分钟完成。`--exit-status` 在失败时返回非零，便于判断结果。

### 5. 验证发布结果

```bash
gh release view v1.2.0                          # 查看 Release 概要
gh release view v1.2.0 --json assets --jq '.assets[].name'   # 确认 5 个产物齐全
```

应包含：`service-edge_linux_amd64`、`service-edge_linux_arm64`、`agent_linux_amd64`、`agent_linux_arm64`、`SHA256SUMS`。

页面地址：`https://github.com/phil616/service-edge/releases/tag/v1.2.0`

### 6.（可选）校验构建溯源

```bash
gh attestation verify <下载的二进制> --repo phil616/service-edge
```

## 五、回滚 / 撤销发错的版本

> 删除已发布的 Release/tag 具有破坏性：若他人或自动化已拉取，会造成混乱。**优先发一个新的 PATCH 版本修正**，仅在确属误发且刚发布时才删除。

```bash
# 1) 删除 GitHub Release
gh release delete v1.2.0 --yes

# 2) 删除远端 tag（GitHub）
git push origin --delete v1.2.0

# 3) 删除本地 tag
git tag -d v1.2.0
```

修正后重新按「第四节」发版（可复用同一版本号，或递增到新的 PATCH）。

## 六、常见问题排查

- **推送 tag 后没有触发工作流**
  - tag 名未匹配 `v*`（例如漏了 `v` 前缀）。
  - 误推送到了 `cnb` 而非 `origin`（发版只认 GitHub）。
  - tag 指向的提交 **不包含** `.github/workflows/release.yml`（该提交早于工作流引入）。用 `git log --oneline <tag> -- .github/workflows/release.yml` 确认。
  - 用 `gh run list --workflow=release.yml` 查看是否真的有运行被触发。

- **工作流失败**
  ```bash
  gh run view <RUN_ID> --log-failed     # 只看失败步骤日志
  ```
  常见原因：前端 `make web` 失败、Go 编译错误、`gh release create` 因 tag 已存在同名 Release 而冲突。

- **同名 tag 已存在 Release，需要重跑**
  先按「第五节」删除旧 Release 与 tag，再重新创建并推送 tag。（当前工作流仅由 tag 推送触发，没有手动 `workflow_dispatch` 入口。）

- **`gh` 未登录 / 无权限**
  `gh auth status` 检查；`gh auth login` 重新登录；确认对仓库有写权限。

- **关于 Node 20 弃用告警**
  工作流运行日志中可能出现 “Node.js 20 actions are deprecated” 的提示，**不影响发布**。后续可将 `actions/*` 升级到支持 Node 24 的版本。

---

## 七、发版后

- 发布的是 **新的二进制**。若已部署的 frps/frpc 主机要用上新功能，需要重新部署对应的 Agent（重新执行安装命令或更新二进制），控制面侧重启新版本即可。
- 如把 Agent 下载源指向 GitHub Release 的 `releases/latest/download/agent`，新版本发布后下次安装将自动拉取最新 Agent。

## 附：一次完整发版的命令速览

```bash
# 0. 确认就绪
git status && git fetch origin --tags && git log --oneline origin/main..HEAD

# 1. 打 tag（替换版本号与说明）
git tag -a v1.2.0 -m "v1.2.0 — <变更摘要>"

# 2. 推送触发发版
git push origin v1.2.0

# 3. 跟踪并验证
gh run list --workflow=release.yml --limit 3
gh run watch <RUN_ID> --exit-status
gh release view v1.2.0 --json assets --jq '.assets[].name'
```
